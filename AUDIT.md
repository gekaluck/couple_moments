## 1. Architecture summary (independent read)

Duet is a Next.js 16 App Router app (React 19, TypeScript, Tailwind v4) for two-person "couple spaces". Routing is file-based: auth pages at root, the app shell under `/spaces/[spaceId]/{calendar,memories,activity,settings}`, and event detail at `/events/[eventId]` (notably *outside* the space shell). Pages are server components that fetch via a thin data layer in `src/lib/*` (Prisma 7 → Postgres), and mutations are **inline server actions defined inside page components**, passed down as props to client components — there is no client state library, no fetch-cache layer; freshness comes from `revalidatePath` + `router.refresh()`. Auth is a custom DB-backed session cookie (`cm_session`) with a middleware presence check plus per-query membership checks (`getCoupleSpaceForUser`) — consistently enforced. Integrations: Google Maps/Places (client-side), Google Calendar OAuth + freebusy sync (server-side), Cloudinary unsigned uploads, Sentry. Styling is Tailwind utilities over a large hand-rolled CSS-custom-property token sheet in `globals.css`; mobile gets a dedicated shell (`MobileTopBar`, `BottomTabBar`, FAB, `MobileAgendaView`) while desktop gets a month grid — two parallel UIs toggled with `hidden md:*`.

The code mostly communicates this design well. The thing it does *not* communicate is where mutation logic is supposed to live: the calendar page is simultaneously layout, data assembler, and the app's biggest mutation API (8 server actions, 911 lines).

## 3. Top 5 — if you can only fix five things

1. **F1 — schedule-idea timezone bug.** It silently corrupts event times in a flagship flow, the fix is small (send `timeZoneOffsetStart` from the IdeaCard schedule form and use `parseLocalDateTime` like `handleCreate` does), and every day it ships it creates wrong data you can't easily repair later.
2. **F2 — blocks are read-only on mobile.** One-way doors are the worst mobile bug class. Making agenda block rows link to `?editBlock=` (the modal already exists and works) is a small change that completes a core flow.
3. **F3 — Maps script in the root layout.** Single-line deletion (the lazy loader already exists) for an app-wide TTI win on every page. Best effort-to-impact ratio in the whole list.
4. **F5 — auth hardening.** Rate-limit login (the UI already expects it) and put a minimum on passwords. This is a private-beta app today; do it before the audience grows.
5. **F6 — fix the two muted text tokens.** Two hex values in `globals.css` lift the entire app over WCAG AA. Almost everything users read in metadata is currently below contrast.

(F4, the calendar god component, is deliberately not in the top 5 fixes — it's the top *restructure*, below. Fixing F1 inside it first is fine.)

## 4. Structural recommendations (do these BEFORE batch bug-fixing)

1. **Extract the event/idea form-data parsing into `src/lib/event-form.ts`.** One `parseEventFormData(formData)` / `parsePlaceFields(formData)` used by all four server actions. F1 happened *because* there are four hand-maintained copies; fix the structure before fixing more parsing bugs or you'll fix them 4×.
2. **Move server actions out of page bodies into co-located `actions.ts` files** (`spaces/[spaceId]/calendar/actions.ts`, `events/[eventId]/actions.ts`). This kills the closure-serialization problem (F12) — actions take `spaceId`/`ideaId` as arguments and re-fetch — shrinks `calendar/page.tsx` by ~450 lines, and makes the mutation surface reviewable. Define a shared `GoogleSyncResult` type there (F18).
3. **Unify the event form.** `add-controls.tsx` (create) and `event-edit-modal.tsx` (edit) are the same form — date/end-date/multi-day/time/offsets/tags/place — maintained twice. Parameterize one `EventForm` before any mobile-form polish lands, or polish lands once.
4. **Decide the timezone story once.** Either store a user/space IANA timezone and do all bucketing with it, or move all grouping client-side (the agenda already serializes ISO strings — it's halfway there). F1, F8, and F22 are one decision, not three bugs.
5. **Purge `globals.css` and collapse the token aliases** (F10, F19) before the next visual pass: delete the dead classes, keep one feedback palette, fix the duplicated `.section-title`. Otherwise the next redesign iteration adds a fourth layer of sediment — the file already records three.
6. **Delete the orphans** (F9) in one commit so file search stops lying to contributors.

## 5. What's good (preserve these)

1. **Type discipline is genuinely excellent.** Zero `any` in ~21k lines of app code, Zod validation at every API route via a tidy `api-utils.ts` (`parseOrBadRequest`, typed auth results), and typed wrappers even around the untyped Google Maps globals. Rare at this stage.
2. **Authorization is consistently layered.** Middleware cookie check → `requireUserId()` → `getCoupleSpaceForUser`/membership scoping inside *every* data-layer mutation, plus SSRF-conscious photo-URL validation (DNS lookup in `lib/events.ts`) and `sanitizeHttpUrl` on all outbound place links. No route I read trusts client-supplied IDs without a membership join.
3. **The mobile shell is a real design, not a squeeze.** Dedicated agenda view instead of a cramped month grid, bottom tabs + FAB with `env(safe-area-inset-*)`, bottom-sheet modals, and `text-base sm:text-sm` on inputs to prevent iOS focus-zoom — someone thought about phones deliberately. The gaps (F2, F13, F15) are completeness, not direction.
4. **Accessibility effort above baseline in the right places.** Headless UI dialogs (focus trap, ESC, labelled titles), a fully APG-correct keyboard menu in `space-nav.tsx` (arrow/Home/End/Escape + focus return), `aria-live` on auth errors, a global `prefers-reduced-motion` kill switch, and per-route loading skeletons.
5. **Graceful Google Calendar degradation.** Every sync path returns a structured `{attempted, success, message, info}` result surfaced as a warning toast — local save never fails because Google did, and stale Google links are detected and self-healed (`recovered`). This is the hardest integration in the app and it has the best error contract.
